# Knowledge-on-Demand — 조직 지식이 결정 순간 소환되는가
### AI/LLM로 조직 지식베이스를 진단해 "무엇을 먼저 이어줄지" 결정하는 분석 · 2026-06

> **한 줄:** 조직은 문서를 쌓지만 정작 의사결정·온보딩 순간에 그 지식이 소환되지 않는다. 이 지식을 그래프로 구조화해 **공백·사일로를 진단**하고, 가장 적은 노력으로 소환성을 높일 **문서화/연결 우선순위**를 냈다.

## 문제 (왜 비즈니스 이슈인가)
조직 지식이 사일로·공백 상태로 방치되면 **온보딩 비용↑·의사결정 속도↓**. "어디에 무엇이 있고, 무엇이 안 꺼내지는가"는 측정되지 않은 채 방치된다.

## 방법 (어떻게 풀었나)
1. **파이프라인 검증(개인 코퍼스):** 내 GitHub stars 28건을 수집→프로파일링→지식그래프(노드=도구, 엣지=topic 공유)→"문제→지식 소환" 함수로 구현. 도메인 편중(AI·LLM 31% vs 비즈니스·인과 14%)을 데이터로 진단.
2. **실제 조직 적용:** 같은 파이프라인을 **GitLab Handbook(People/Hiring) 82페이지**에 적용. 페이지=노드, 내부 링크=엣지로 그래프화.
3. **AI투명성:** 구조지표(고립·stub·허브)는 결정적으로 계산. LLM/판단이 개입한 부분은 **샘플 페이지를 직접 열어 휴먼검증**하고, 입증된 것만 결론에 사용.

## 진단 (측정값)
- **orphan(들어오는 링크 0) 21건 = 25.6%**, 완전 고립 8건. → People/Hiring 지식의 **1/4이 핸드북 내에서 소환되지 않음**.
- **묻힌 자산:** 신입 가이드 *TaNewKi Tips*(2,554단어)·*Mentoring*·*Internal Hiring Process*·채용운영 최대문서 *Coordinator*(20,058단어)·*Anti-Harassment 정책*이 전부 orphan. 콘텐츠는 충실한데 **링크가 끊겨** 안 떠오름.
- **온보딩 시나리오:** 신입 첫 주 핵심 페이지의 **절반이 링크로 소환 불가**.

## 결정 (그래서 무엇을)
1. **연결 보강 최우선(최저비용·최대효과):** 묻힌 자산을 허브(Onboarding·Hiring·People Group)에서 링크 → 즉시 소환. *"새로 쓰기"가 아니라 "이어주기"가 해법.*
2. **진입점 수리:** 산문 참조만 있는 섹션 인덱스(Time Off)를 실제 링크로 → 고립 해소.
3. **컴플라이언스 시급:** orphan 상태인 정책 문서(Anti-Harassment 등) 연결 1순위 승격.
4. **보강/통합:** stub·고립 페이지 통합.

## 차별점·판단력
- **AI에 안 끌려감:** LLM 추출을 그대로 믿지 않고 샘플 휴먼검증으로 신뢰를 입증한 뒤에만 결론에 사용.
- **드리프트 통제:** "이 작업이 취업 산출물로 떨어지나"를 기준(`docs/NORTH_STAR.md`)으로, 개인 stars 그래프는 *파이프라인 검증*에서 동결하고 비즈니스 가치가 있는 조직 코퍼스로 이동(`DECISION_LOG` D-010). 도구 과over-building을 의식적으로 차단.
- **결정 지향:** 모든 분석을 "그래서 무슨 결정"으로 종료.

## 프로젝트 회고 (STAR 요약)
- **S/T:** 조직 지식이 결정 순간 소환되는지 데이터로 진단하고 싶었다.
- **A:** GitHub stars로 수집→그래프→소환 파이프라인을 검증한 뒤 GitLab Handbook에 적용. 링크 그래프로 orphan·고립·허브를 결정적으로 계산하고, 샘플을 직접 열어 검증.
- **R:** People/Hiring 82p 중 25.6%가 소환 불가 orphan임을 밝히고, "문서 부족이 아니라 연결 부족"이라는 진단으로 **최저비용 연결 보강 우선순위**를 제시. AI 출력은 검증 후에만 사용하는 규율을 지킴.
