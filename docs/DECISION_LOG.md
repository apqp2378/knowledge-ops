# DECISION_LOG — knowledge-ops에 오기까지의 의사결정·변화 기록

## 0. 출발점 — 두 개의 30일 플랜
- **self-da 플랜(v2):** 준비를 *자동화 메타 시스템*으로. NL→SQL 가상 해커톤, da_toolkit 3종(nl2sql/csv2html/streamlit), gap matcher 등.

- 두 플랜의 공통 병폐: **작은 도구·산출물이 너무 많음(넓고 얕음).**

## 1. 실행 — Module A에 집중
self-da의 **Module A(채용공고 분석 파이프라인)**를 구현: parser → clean → rule classifier → skill extractor + 테스트 + 스키마 문서.
- **D-006 (판단력 사례):** JD 코퍼스가 작고 동질적 → 비지도(BERTopic) 부적합. **규칙 기반 분류로 전환**, BERTopic은 검증용 강등. *AI가 제안한 방법을 데이터 근거로 뒤집은 사례.*

## 2. 드리프트 자각
- Module A는 **어느 플랜에도 핵심이 아니었다.** 플랜의 채용공고 분석은 "한 세션 → README 한 장"이 전부인데, 테스트까지 갖춘 소프트웨어로 과대 성장.
- 증상: "빙빙 도는 느낌, 와닿지 않음, 불필요한 작업이 많음."

## 3. 원인 진단 (실패모드 3)
1. Cowork의 **한 단계씩 꼬리물기** → 길을 잃음.
2. 매 턴 **새 프레임 생성**(4트랙 → 차별화 → STAR) → 또 다른 빙빙 돌기.
3. **수단(코드/파이프라인)이 목적(핵심 산출물)을 잡아먹음.**
- 재정렬: 지금 목표 = **깊은 플래그십 케이스 1개**. 방향 = **"AI/LLM로 실제 비즈니스 데이터 문제를 푸는 분석."**

## 4. 플래그십 케이스 탐색 (수렴 과정)
| 후보 | 평가 | 기각/채택 사유 |
|---|---|---|
| Olist 이커머스 + LLM 리뷰 | 기술적으론 깔끔 | "남의 데이터"라 진정성·완주 약함 → 또 드리프트 위험 |
| C안 원안: YouTube 학습노트 | 진정성 강 | 개인 학습 문제라 비즈니스성·의사결정 약함 |

## 5. 결정 (현재)
- **D-007:** 두 30일 플랜 + Module A 전부 **파킹**.
- **D-008:** 플래그십 = **Knowledge-on-Demand(조직 KM 진단)**, **새 레포 `knowledge-ops`**.
- 포트폴리오 = 리워드광고(이커머스 도메인) + KoD(AI/LLM 차별점). 2종 역할 분담.

## 6. 배운 점
- **넓고 얕게 < 좁고 깊게:** 도구 N개보다 깊은 케이스 1개.
- **드리프트 경계:** "이 작업이 핵심 산출물로 떨어지나?"를 매 작업마다 묻는다. 수단이 목적을 잡아먹지 않게.
- **진정성·완주:** 와닿는 *내 문제*라야 끝까지 간다. 기술적 완벽함보다 우선.
- **AI에 끌려가지 않기:** D-006처럼 AI 제안을 데이터로 검증·기각할 수 있어야 한다.

## 8. 코퍼스 성격 변경 (06-24)
- **D-010:** 개인 코퍼스에 *기존 star가 없음*이 확인됨(수집 0건) → Phase 1을 "이미 쌓인 별 진단"에서 **"시드 코퍼스 구축 → 진단"**으로 변경. 4도메인(데이터분석·시각화·AI/LLM·이커머스) 시드 star 후 ponytail(노벨티)·포크 정리.
  - EDA 초점도 '방치기간'에서 **'커버리지·공백·중복'**으로 이동: `starred_at`이 전부 동일자라 방치 지표는 무의미. **"저장만 하고 안 씀"은 비유였고, 본질은 쌓인 지식의 그래프화·소환·공백진단·정리결정임을 재확인**(방치 페인 입증에 집착하지 않기).
  - 근거: 본질은 코퍼스 출처가 기존/신규든 동일하게 성립. 1차 진단 = AI·LLM 편중·비즈니스/인과 공백 → 보강 결정으로 종료.
  - 산출: `src/collect_stars.py`(수집), `src/profile_stars.py`(프로파일 EDA), `src/build_graph.py`(지식그래프 + 문제→지식 소환). 데이터·그래프는 CSV/JSON으로 외부화(재현 가능).
