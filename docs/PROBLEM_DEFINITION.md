# 문제 정의 (잠금) — knowledge-ops / Knowledge-on-Demand
### 조직 KM 버전 · v1.1 (2026-06-24)

> 이 문서는 개념 잠금이다. 이후 작업은 여기서 벗어나지 않는다.
> 목표: 데이터 분석가 신입 취업. 차별점 = "AI/LLM으로 실제 문제를 푸는 분석가".

## 1. 문제 (개인 페인 → 조직 문제)
**개인(원천 동기):** 문제 해결에 도움될 것을 **GitHub 별(star)·좋아요·나중에 볼 영상**으로 저장하지만, 막상 그 상황이 오면 ① 저장한 걸 **까먹고**, ② **어디 있는지 못 찾고**, ③ **어떻게 적용할지 막막**하다.

**조직(비즈니스 일반화):** 조직도 똑같다 — 문서·회의록·위키·교육자료를 쌓지만, 정작 의사결정·온보딩 순간에 **그 지식이 소환·적용되지 않는다.** 지식이 **사일로·중복·공백** 상태로 방치돼 의사결정 속도와 온보딩 비용에 손해.

## 2. 한 줄 정의
저장된 지식을 AI로 구조화해 **지식그래프**로 축적하고, 그 그래프를 데이터로 분석해
① 문제에 맞는 지식 **소환**, ② **공백·중복·사일로 진단**, ③ 채울/문서화/정리 **우선순위 결정**을 낸다.

## 3. 분석 질문 5개
- Q1. 지식베이스는 어떤 개념·주제로 구성되고, **어디가 편중/공백/사일로**인가? (그래프 위상·커버리지)
- Q2. 특정 문제(예: "스트리밍 파이프라인 구축", "온보딩")가 주어지면, 그래프에서 **어떤 지식/도구가 소환되고 어떻게 적용**되나? (문제→지식 매핑)
- Q3. 저장 항목의 **신뢰·성숙·최신성**과 **중복(같은 use-case 다수)**은? (쓸 지식 가려내기)
- Q4. 무엇이 **중복 과잉 / 미충족(공백)**인가? (중복·갭)
- Q5. 진단 결과로 **무엇을 먼저 정리/문서화/폐기**해야 하나? (의사결정)

## 4. 데이터 (확정)
- **개인 코퍼스(파이프라인 검증·진정성) = 내 GitHub starred repos.**
  GitHub API로 메타(설명·README·topics·언어·스타·최근커밋) + README 텍스트 → "별만 찍고 잊은 도구"를 문제→도구로 소환.
- **조직 코퍼스(비즈니스 KM 진단) = GitLab Handbook**(공개 대형 핸드북, markdown) → 지식 갭·중복·사일로 진단 → 문서화 우선순위.
- **(확장·scope 제외) YouTube 저장 영상** — 같은 파이프라인 적용 가능. 폭주 방지로 scope-1엔 미포함.
- 산출 그래프: 노드(개념/도구)·엣지(관계)·속성(출처·신뢰/성숙·use-case·중복그룹) → CSV/JSON **외부화(재현 가능)**.

## 5. AI 엔진
LLM이 원문(README·핸드북 문단) → atomic 개념/도구·관계·**use-case·신뢰/성숙등급**·중복 후보 추출 → 그래프(+Obsidian 표현).
**검증:** 샘플 휴먼체크로 LLM 분류 정확도 입증(= "AI에 안 끌려간" 증거).

## 6. 분석 → 의사결정 (평가받는 산출물)
- 그래프 분석: 중심성(허브)·클러스터(주제)·**고립/저밀도(공백)**·중복 클러스터
- 신뢰/중복: 성숙·최신성 분포, 동일 use-case 중복
- **문제→지식 매핑:** 의사결정 시나리오 N개에 대한 소환·적용 가이드 (핵심 차별 산출물)
- 의사결정: "정리/문서화/폐기 우선순위 Top N + 근거"

## 7. 성공·품질 기준 (그럴듯 방지)
재현 가능(데이터·그래프 외부화) · 진짜 숫자만, 인사이트는 측정값 앵커 · 모든 분석은 "그래서 무슨 결정"으로 종료 · AI가 한 것 vs 내가 검증한 것 구분 · 모든 선택에 "왜" 답변 가능.

## 8. 스코프 가드
산출물은 의사결정 1~2개로 한정 · 코퍼스는 **GitHub stars(개인) + GitLab Handbook(조직) 2개로 고정**(YouTube 등 추가 금지) · Obsidian은 수단, 평가 대상은 분석 · 기능 무한확장 금지.
